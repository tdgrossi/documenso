# Symptoms

## Expected Behavior

When a recipient completes a signing field (signature, date, text, initials, checkbox), the field border should turn grey (`ring-neutral-400`, computed `--tw-ring-color: rgb(163, 163, 163)`) to visually distinguish filled fields from empty ones.

## Actual Behavior

Completed fields retain their full recipient color (e.g., green/blue/etc.) — the grey border does not appear. The `ring-neutral-400!` class was added to `FieldRootContainer` conditionally on `field.inserted`, but it's not being applied visually.

## Error Messages

None — no console errors. The CSS class is likely present in the DOM but being overridden by higher-specificity or later-cascade styles.

## Timeline

Phase 2 (Field Border UX) was implemented and deployed to a test environment. The plan added a single-line change: `'ring-neutral-400!': field.inserted` inside the `cn({...})` call in `FieldRootContainer` (`packages/ui/components/field/field.tsx:127`).

## Reproduction

1. Deploy the Phase 2 changes to a test environment
2. Open a signing page for a document with pending fields
3. Sign/complete any field
4. Observe the field border — it shows the recipient's color (e.g., green) instead of grey (`neutral-400`)

# Resolution

root_cause: "The `!important` modifier was used as a suffix (`ring-neutral-400!`) instead of a prefix (`!ring-neutral-400`). Tailwind v3 only recognizes `!` as a prefix modifier (`!ring-neutral-400`), which generates `.\!ring-neutral-400 { ... !important }`. The suffix syntax `ring-neutral-400!` is silently ignored by Tailwind's content scanner — no CSS class is generated. Additionally, `twMerge` v1.14 treats `ring-neutral-400!` as a valid ring utility and removes competing ring classes (`ring-gray-200`, `ring-recipient-green`), leaving the element with class `ring-neutral-400!` in the HTML but with no corresponding CSS rule, causing the field to render with no ring color (falling back to default)."

fix: "Changed `'ring-neutral-400!': field.inserted` to `'!ring-neutral-400': field.inserted` in `packages/ui/components/field/field.tsx:127`. The `!` modifier must be a prefix in Tailwind v3 to be recognized. With the fix, Tailwind generates `.\!ring-neutral-400` with `!important` on both `--tw-ring-opacity` and `--tw-ring-color`, which overrides the `ring-recipient-green` class. Both classes remain in the DOM (twMerge doesn't remove `ring-recipient-green` when `!ring-neutral-400` is passed), but the `!important` ensures the grey color wins the cascade."
